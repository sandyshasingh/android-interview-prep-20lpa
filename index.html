Kotlin & Coroutines
Explain how Kotlin coroutines work internally — what is a Continuation? ▶
A coroutine is implemented via Continuation Passing Style (CPS) transformation by the Kotlin compiler. When you mark a function as suspend, the compiler rewrites it into a state machine. The function receives an implicit Continuation<T> parameter which is essentially a callback.

Each suspension point becomes a state in that state machine. When the coroutine suspends (e.g., waiting for a network response), it saves its local variables and current state index into the Continuation object and returns — freeing the thread. When the result is ready, the Continuation is resumed, the state machine jumps back to the correct state, and execution continues.

This is why coroutines are lightweight — they don't block threads; they save state on the heap and yield the thread back to the pool. 
// Simplified compiler output concept:
fun fetchData(continuation: Continuation): Any {
    when (continuation.state) {
        0 -> { /* initial state */ continuation.state = 1; return SUSPEND }
        1 -> { /* resumed after suspend */ return result }
    }
}
StateFlow vs SharedFlow — when would you use each? ▶
StateFlow is a hot flow that always holds a current value (like LiveData). It requires an initial value, replays the latest emission to new collectors, and only emits when the value actually changes (uses equals()). Use it for UI state — e.g., a loading/success/error state object.

SharedFlow is more flexible — no initial value required, configurable replay cache (0 to N), and emits every value regardless of equality. Use it for one-time events like navigation triggers, snackbar messages, or analytics events.

Rule of thumb: StateFlow for state, SharedFlow for events. 
// ViewModel
val uiState: StateFlow<HomeState> = _uiState.asStateFlow()  // state
val events: SharedFlow<UiEvent> = _events.asSharedFlow()   // events
What is structured concurrency and why does it matter? ▶
Structured concurrency means coroutines have a defined lifetime tied to a scope. When a scope is cancelled (e.g., ViewModel is cleared → viewModelScope is cancelled), all coroutines launched within it are automatically cancelled too. This prevents coroutine leaks.

Key rules: a parent coroutine waits for all its children to complete. If a child fails with an exception, the parent and all siblings are cancelled (unless using SupervisorJob). With SupervisorJob, child failures are isolated — other children continue.

This is why you should always use viewModelScope or lifecycleScope instead of GlobalScope — GlobalScope has no structure and creates coroutine leaks.

Architecture
How does ViewModel survive configuration changes? ▶
ViewModel is stored in a ViewModelStore, which is retained by the system across configuration changes (like screen rotation). When an Activity is recreated, the new instance receives the same ViewModelStore from the ViewModelStoreOwner, and thus gets the same ViewModel instance back.

The Activity's NonConfigurationInstance mechanism holds the ViewModelStore alive during the recreation. Only when the Activity is truly finishing (user presses back, or isFinishing returns true) is the ViewModelStore cleared and ViewModel.onCleared() called.

This means you should never hold references to Context, Views, or Activity in a ViewModel — they'll be destroyed and recreated, but the ViewModel lives on, causing leaks.
How would you architect an offline-first feature for Classplus (e.g., downloading course videos)? ▶
Layers involved:

1. Download Management: Use WorkManager with a DownloadWorker that handles chunked downloads. WorkManager ensures the download survives process death and respects system constraints (network available, battery not low).

2. Storage: Save video files to app-specific external storage (getExternalFilesDir()). Track download state and file path in Room DB.

3. Repository layer: Before fetching from network, always check Room for cached content. Emit Room data first (cache), then fetch network and update Room, which automatically emits via Flow to the UI.

4. Playback: Use ExoPlayer with a custom DataSource.Factory — serve local file if downloaded, otherwise stream with adaptive bitrate.

5. DRM: Use ExoPlayer's DefaultDrmSessionManager with Widevine for protected content.
What is dependency inversion and how does DI support it? ▶
Dependency Inversion Principle (the D in SOLID) states: high-level modules should not depend on low-level modules — both should depend on abstractions (interfaces).

In Android: instead of your ViewModel directly instantiating a UserRepository class, it depends on a IUserRepository interface. The concrete implementation is injected at runtime via Hilt/Dagger.

Benefits: your ViewModel is testable without a real network or database — you inject a FakeUserRepository in tests. You can swap implementations (e.g., mock vs real) without changing the ViewModel.

Hilt achieves this via compile-time code generation — it creates a dependency graph at build time, so there's no runtime reflection overhead (unlike older approaches).

Jetpack Compose
How does Compose decide what to recompose, and how do you optimize it? ▶
Compose uses smart recomposition — it only recomposes composables whose inputs (state reads) have changed. The Compose compiler inserts tracking calls around state reads. When a State<T> changes, only the composables that read that specific state are scheduled for recomposition.

Optimization techniques:
1. Stable types: Mark data classes with @Stable or @Immutable to tell the compiler parameters won't change unexpectedly — enabling skippability.
2. derivedStateOf: Wraps a computation that derives from other state, so it only recomputes when its inputs change.
3. key(): In LazyColumn, use stable unique keys so Compose can reuse composables instead of re-creating them on list updates.
4. Avoid reading state high in the tree — push state reads down to the leaf composables that actually need them, limiting recomposition scope.
LaunchedEffect vs rememberCoroutineScope — when to use which? ▶
LaunchedEffect(key): Launches a coroutine tied to the composable's lifecycle. It starts when the composable enters composition and is cancelled when it leaves. Re-launches when the key changes. Use it for work that should run automatically in response to a state change — e.g., fetching data when a userId changes, playing an animation on appearance.

rememberCoroutineScope(): Returns a CoroutineScope tied to the composable's lifecycle, but you launch coroutines manually from event handlers. Use it when you need to trigger a coroutine from a user interaction — e.g., inside an onClick lambda.

Key rule: Automated side effects → LaunchedEffect. User-triggered coroutines → rememberCoroutineScope. 
// LaunchedEffect — auto-triggered
LaunchedEffect(userId) { viewModel.load(userId) }

// rememberCoroutineScope — user-triggered  
val scope = rememberCoroutineScope()
Button(onClick = { scope.launch { doSomething() } }) { ... }
